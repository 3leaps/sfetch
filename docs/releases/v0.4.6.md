## Summary
Private GitHub repo asset downloads now succeed when the user has visibility, and token resolution learns `GH_TOKEN` plus a `--token-env <NAME>` escape hatch for the wrong-scope-PAT case.

## Highlights

### Private-repo asset download fix
- The browser download URL (`https://github.com/<o>/<r>/releases/download/...`) returns 404 for private-repo assets even with a valid Bearer token. When a token is available, sfetch now uses the GitHub API asset endpoint (`https://api.github.com/repos/<o>/<r>/releases/assets/<id>`) with `Accept: application/octet-stream`, follows the resulting 302 to a signed S3 URL, and writes the asset to disk.

### Token resolution
- New default precedence: `SFETCH_GITHUB_TOKEN` → `GH_TOKEN` → `GITHUB_TOKEN`. `GH_TOKEN` is what `gh auth login` populates by default, so it now works without manual export.
- `--token-env <NAME>` reads the token from a named env var. Useful when the ambient `GITHUB_TOKEN` cannot see a sibling org's private repo and a separately-scoped PAT is loaded under a different name. Hard-fails when the named var is empty rather than silently falling back.
- sfetch deliberately refuses `--token <value>` to keep secrets out of `~/.bash_history`, `ps`, and `set -x` CI logs.

### Defense-in-depth on redirects
- Every sfetch GitHub HTTP request now installs a custom `CheckRedirect` that strips `Authorization` on hops to non-trusted hosts, even when Go's stdlib same-domain rule would have preserved the header. Covers the `github.com` → S3 hop, plus any redirect to a non-github host.

### Auth-failure error UX
- 401/404 from an asset download names the env-var source the token came from (never the value) and prescribes `--token-env <YOUR_VAR_NAME>` as the recovery path.

## Breaking Changes
None. The new `--token-env` flag is additive; the default token chain is a superset of v0.4.5 (now also reads `GH_TOKEN`).

## Migration
No migration required for existing public-repo flows. For private-repo downloads:

```bash
# Before v0.4.6: silent 404 on private repos
sfetch --repo myorg/private-tool --latest

# v0.4.6 with default chain (works if GITHUB_TOKEN is set and scoped):
sfetch --repo myorg/private-tool --latest

# v0.4.6 with a differently-scoped PAT:
export PRIVATE_REPO_PAT=ghp_xxx_with_repo_access
sfetch --repo myorg/private-tool --latest --token-env PRIVATE_REPO_PAT
```

## Validation
- `go test -race ./...`
- `make prepush`
- httptest integration test asserts: API asset endpoint is hit with `Accept: application/octet-stream` + Bearer token, redirect to signed URL succeeds, no `Authorization` header reaches the signed-URL host.

## Examples

```bash
# Default chain (uses SFETCH_GITHUB_TOKEN, GH_TOKEN, or GITHUB_TOKEN)
sfetch --repo myorg/private-tool --latest --asset-match linux-amd64

# Explicit scoped PAT for one call
sfetch --repo otherorg/private-tool --latest --token-env OTHER_ORG_PAT
```

CI snippet:

```yaml
- name: Install sibling-org tool
  env:
    PRIVATE_REPO_PAT: ${{ secrets.PRIVATE_REPO_PAT }}
  run: |
    sfetch --repo otherorg/cli --latest \
           --asset-match linux-amd64 \
           --dest-dir "$HOME/.local/bin" \
           --token-env PRIVATE_REPO_PAT
```
